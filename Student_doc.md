# SYSTEM DESCRIPTION:

The system is a distributed and fault-tolerant seismic analysis platform designed to detect and classify seismic events from real-time sensor streams provided by an external simulator.

The simulator exposes geographically distributed seismic sensors, each producing continuous time-domain measurements through WebSocket streams. It also exposes a shared control stream used to simulate failures of processing replicas by issuing shutdown commands.

To comply with the mission constraints, the neutral region hosts only lightweight routing logic. For this reason, a custom broker service is deployed as a fan-out component: it discovers available sensors, subscribes to their streams, and redistributes each measurement to multiple replicated processing services without performing any form of intelligence processing.

Processing services are replicated and deployed as independent backend nodes. Each replica maintains an in-memory sliding window for every sensor, performs FFT-based frequency-domain analysis, extracts the dominant frequency, classifies events according to the required frequency bands, and persists detected events into a shared PostgreSQL database.

The persistence layer is duplicate-safe. Equivalent detections generated by multiple replicas or overlapping windows are controlled through deterministic event identifiers, database-level conflict handling, and an additional in-memory suppression policy.

A gateway service provides a single backend entry point for the frontend dashboard. The dashboard supports near real-time event monitoring, historical exploration, filtering, pagination, and system status visualization.

The overall platform remains operational under partial failure conditions. When one processing replica receives a shutdown command from the simulator control stream, it terminates itself, while the broker continues forwarding measurements to the remaining reachable replicas and the monitoring capabilities remain available.

### PORTS:
8001:8001

### DESCRIPTION:
The broker-service container is the fan-out component of the architecture. It discovers all available sensors from the simulator, opens one WebSocket connection per sensor, receives time-domain measurements, and forwards each measurement to the configured processing replicas. The broker does not perform any FFT, classification, or event-detection logic.

### PERSISTENCE EVALUATION
This container does not persist application data. It only handles transient live measurements and delivery queues.

### EXTERNAL SERVICES CONNECTIONS
The broker connects to the external simulator through REST and WebSocket, and connects to the processing replicas through internal HTTP calls.

### MICROSERVICES:

#### MICROSERVICE: broker
- TYPE: backend
- DESCRIPTION: Discovers simulator sensors, consumes their streams, and forwards each measurement to the processing replicas.
- PORTS: 8001
- TECHNOLOGICAL SPECIFICATION:
The microservice is developed in Python and uses FastAPI.
It uses the following libraries and technologies:
    - FastAPI: provides the HTTP service and lifecycle hooks
    - Uvicorn: runs the ASGI application
    - websockets: consumes real-time sensor streams
    - requests: performs REST calls to the simulator and HTTP forwarding to replicas
    - asyncio: coordinates background sensor consumers and worker queues
- SERVICE ARCHITECTURE:
The service is realized with:
    - a FastAPI entrypoint with startup logic
    - a simulator client module for sensor discovery
    - one asynchronous consumer per sensor WebSocket stream
    - multiple internal worker queues used to preserve per-sensor ordering
    - a distributor module that forwards each measurement to the configured processing replicas
    - replica reachability tracking with temporary exclusion and retry

- ENDPOINTS:

    | HTTP METHOD | URL | Description | User Stories |
    | ----------- | --- | ----------- | ------------ |
    | GET | /health | Returns broker health information including connected sensors and configured replicas | 10, 11 |
    | GET | /runtime-info | Returns queue statistics, dropped measurements, and runtime distribution information | 11, 20 |

### DESCRIPTION:
The processing-service container is instantiated multiple times as replicated processing nodes. Each replica receives measurements from the broker, maintains in-memory sliding windows per sensor, performs FFT-based dominant-frequency extraction, classifies seismic events according to the required frequency bands, applies in-memory duplicate suppression, stores events in PostgreSQL with conflict-safe insertion, and listens to the simulator control stream to terminate upon shutdown commands.

    - an in-memory event deduplicator to reduce repeated detections on overlapping windows
    - a persistence module with deterministic event-id generation and conflict-safe insertion
    - a control listener module that self-terminates the replica when the simulator emits a shutdown command
